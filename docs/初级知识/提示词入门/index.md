# 提示词入门

提示词是你和大语言模型协作的接口。好的提示词不是"咒语"，而是把任务、背景、约束、输出格式和判断标准说清楚。

## 什么是提示词？

提示词（Prompt）是你发给大语言模型的输入文本，它决定了模型的理解方向和输出质量。提示词可以是：
- 一个简单问题："什么是机器学习？"
- 一段详细指令："用通俗语言解释机器学习，举 3 个日常例子，每个例子不超过 50 字"
- 包含示例的模板："把以下格式应用到新数据上..."

### 为什么提示词重要？

| 差的提示词 | 问题 | 好的提示词 | 效果 |
|-----------|------|-----------|------|
| "写个摘要" | 模型不知道多长、什么风格 | "用 3 句话总结这篇论文，突出主要发现和方法" | 长度可控、重点明确 |
| "帮我优化代码" | 没说优化什么、标准是啥 | "优化这段 Python 代码的运行速度，减少循环次数，保持可读性" | 目标清晰、可验证 |
| "翻译成英文" | 语境不明 | "把这段产品描述翻译成英文，用正式商务语气，面向B2B客户" | 风格准确、场景适配 |

## 提示词的基本结构（6要素框架）

好的提示词通常包含这些元素（根据任务可选择使用）：

```text
【角色 Role】你希望模型以什么身份处理任务
【任务 Task】要完成什么，目标是什么
【背景 Context】必要上下文、参考资料、历史信息
【约束 Constraints】不能做什么、需要注意什么、边界在哪
【格式 Format】输出结构、长度、样式
【标准 Criteria】什么算好答案、如何评判质量
```

### 实际案例对比

**基础版（缺少元素）：**
```text
帮我写个产品介绍
```
→ 模型不知道产品是什么、面向谁、多长、什么风格

**完整版（6要素）：**
```text
【角色】你是一名面向企业客户的 SaaS 产品文案撰写专家。
【任务】为我们的 AI 数据分析平台撰写产品介绍。
【背景】产品名称：DataInsight Pro；核心功能：自动数据清洗、可视化看板、预测分析；目标客户：中小型企业的数据团队。
【约束】不使用"革命性""颠覆性"等夸张词汇；避免复杂技术术语；重点突出易用性和性价比。
【格式】分为三段：第一段一句话概述价值，第二段列举 3 个核心优势（每个 30 字左右），第三段给出典型使用场景。
【标准】读者能在 30 秒内理解产品核心价值，并产生"想试试"的兴趣。
```
→ 输出高度可控、风格准确、内容聚焦

## 从零开始的提示词演进

让我们看一个真实任务如何从模糊指令迭代到高质量提示词：

**任务：** 分析一段客户反馈

### 第 1 版（过于简单）
```text
分析这条客户反馈
```
**问题：** 分析什么？深度如何？输出什么格式？

### 第 2 版（加入任务说明）
```text
分析这条客户反馈，告诉我客户的主要问题
```
**问题：** 主要问题可能有多个，要全部列出还是最重要的一个？要不要给解决建议？

### 第 3 版（明确输出结构）
```text
分析以下客户反馈：
[反馈内容]

请输出：
1. 客户的核心诉求（一句话）
2. 提及的具体问题（列表）
3. 情绪评估（正面/中性/负面，简要说明原因）
```
**改进：** 结构清晰，但还缺少对"分析深度"和"如何处理歧义"的说明

### 第 4 版（加入约束和标准）
```text
你是一名客户成功经理，擅长从反馈中识别真实需求。

请分析以下客户反馈：
[反馈内容]

输出格式：
1. 核心诉求：用一句话概括客户最想解决的问题
2. 具体问题点：列出反馈中提到的所有问题（如有多个，按影响程度排序）
3. 情绪判断：正面/中性/负面，并引用原文关键词作为依据
4. 建议优先级：紧急/重要/可延后，基于问题严重程度和客户语气判断

约束：
- 只分析反馈中明确提到的内容，不要推测未提及的需求
- 如果反馈模糊或矛盾，在相应部分注明"需进一步确认"
- 引用原文时用引号标注

标准：
- 核心诉求提炼准确，能指导后续行动
- 问题点不遗漏关键信息
- 情绪判断有文本支撑，不主观臆断
```
**效果：** 输出稳定、可直接用于工作流程、减少二次澄清

## 实用技巧速查表

| 技巧 | 作用 | 示例 |
|------|------|------|
| **明确任务** | 避免模型猜测你的目的 | ❌ "帮我看看这段代码" → ✅ "检查这段代码是否有安全漏洞" |
| **提供背景** | 减少错误假设 | "这是面向医疗行业的API，需符合HIPAA合规要求" |
| **限定范围** | 防止输出发散 | "只讨论技术实现，不涉及商业模式" |
| **指定格式** | 便于后续处理 | "用 JSON 格式输出，字段包括 title, summary, tags" |
| **给示例** | 让模型模仿风格 | 提供 1-3 个符合期望的输出样例 |
| **要求自检** | 提醒模型检查遗漏 | "输出前检查是否所有约束都已满足" |
| **分步执行** | 降低复杂任务出错率 | "第一步：提取关键信息；第二步：分析；第三步：总结" |
| **使用分隔符** | 清晰区分指令和内容 | 用 `###`、`"""` 或 `---` 分隔不同部分 |

## 不同任务类型的提示词模板

### 1. 信息提取

```text
从以下文本中提取关键信息：
[文本内容]

需要提取的字段：
- 人名：
- 时间：
- 地点：
- 事件：

要求：
- 如果某字段在文本中未提及，填写"未提及"
- 时间统一格式为 YYYY-MM-DD
- 如有歧义，列出所有可能性
```

### 2. 内容改写

```text
请将以下内容改写为[目标风格]：
[原始内容]

改写要求：
- 保持原意不变
- 调整语气为[正式/轻松/专业/通俗]
- 目标读者：[具体描述]
- 长度控制在[X]字左右

不要改动：
- 专有名词
- 数据和引用
```

### 3. 创意生成

```text
背景：[项目/产品/场景描述]
目标受众：[用户画像]

请生成 5 个[标语/标题/方案]，要求：
- 每个不超过[X]字
- 风格：[描述期望风格]
- 必须包含关键词：[列出关键词]
- 避免使用：[禁用词列表]

输出格式：
1. [创意内容] - 简短说明（为什么这个好）
2. ...
```

### 4. 问题诊断

```text
角色：你是[领域]专家
场景：用户遇到以下问题

问题描述：
[详细描述]

已尝试的解决方法：
[列举]

请诊断可能的原因，并给出解决建议：
1. 最可能的原因（附理由）
2. 其他可能原因（2-3个）
3. 排查步骤（优先级排序）
4. 临时解决方案
5. 长期建议

约束：
- 基于描述的信息，不臆测未提及的细节
- 如果信息不足，明确指出需要补充哪些信息
```

### 5. 数据分析

```text
以下是一组数据：
[数据内容]

请分析并输出：
1. 数据概览（记录数、时间范围、主要维度）
2. 关键发现（3-5条，按重要性排序）
3. 异常点（如有）
4. 趋势判断
5. 建议（基于数据支持的可操作建议）

输出约束：
- 所有结论必须在数据中有依据
- 用百分比或具体数字量化描述
- 如需图表，说明应绘制的图表类型和维度
```

## 输出格式控制详解

### JSON 输出

当输出需要程序处理时，JSON 是最佳选择：

```text
请分析以下产品评论并输出 JSON，不要有任何解释文字。

评论：[内容]

输出格式：
{
  "sentiment": "positive/neutral/negative",
  "rating_estimate": 1-5,
  "key_points": ["要点1", "要点2", ...],
  "mentioned_features": ["功能A", "功能B", ...],
  "concerns": ["问题1", "问题2", ...]
}

要求：
- sentiment 必须是三个枚举值之一
- key_points 不超过 5 条，每条不超过 30 字
- 如果某字段无内容，返回空数组 []
```

### Markdown 表格

适合对比和展示结构化信息：

```text
请对比以下三个方案，用 Markdown 表格输出：

方案A：[描述]
方案B：[描述]
方案C：[描述]

对比维度：
- 成本
- 实施难度
- 预期效果
- 风险

表格格式：
| 方案 | 成本 | 实施难度 | 预期效果 | 风险 |
|------|------|---------|---------|------|
| A | ... | ... | ... | ... |

表格后附简短推荐理由（50字内）
```

### 分段输出

适合长文档或报告：

```text
请按以下结构输出分析报告：

## 一、执行摘要
[3句话概括核心发现和建议]

## 二、背景情况
[问题来源、涉及范围]

## 三、详细分析
### 3.1 [维度1]
[分析内容]
### 3.2 [维度2]
[分析内容]

## 四、结论与建议
[行动建议，优先级排序]

要求：
- 每段不超过 150 字
- 使用二级和三级标题
- 关键数据用粗体标注
```

## 常见采样参数入门

提示词之外，采样参数也直接影响输出。初学者至少理解这几个：

| 参数 | 作用 | 调整直觉 | 推荐值 |
|------|------|----------|--------|
| **temperature** | 控制随机性 | 越高越发散（创意），越低越确定（事实） | 事实/代码: 0-0.3<br>创意: 0.7-1.0 |
| **top_p** | 限制采样候选范围 | 和 temperature 二选一调整 | 0.9-0.95 |
| **max_tokens** | 限制输出长度 | 防止跑题和成本失控 | 按需设定，留20%余量 |
| **frequency_penalty** | 惩罚重复词汇 | 减少啰嗦和循环 | 0.3-0.7 |
| **presence_penalty** | 鼓励话题多样性 | 让输出更丰富 | 0.3-0.6 |
| **seed** | 固定随机性 | 需要可复现结果时使用 | 任意整数 |

### 参数组合建议

| 任务类型 | temperature | top_p | 备注 |
|---------|-------------|-------|------|
| 事实问答、数据分析 | 0-0.2 | 0.1 | 追求准确性 |
| 代码生成 | 0-0.3 | 0.1 | 避免语法错误 |
| 技术文档、教程 | 0.3-0.5 | 0.9 | 平衡准确与可读 |
| 产品文案、营销 | 0.7-0.9 | 0.95 | 需要吸引力 |
| 创意写作、头脑风暴 | 0.8-1.0 | 0.95 | 追求多样性 |

> **实验建议**：固定提示词，只改参数，对比 3-5 次输出，找到最适合你任务的配置。

## 提示词不适合解决什么

提示词不是万能的。以下问题通常需要其他方法：

| 问题 | 为什么提示词不够 | 更合适的方法 |
|------|-----------------|-------------|
| 模型不知道私有资料 | 上下文窗口有限 | RAG（检索增强）或上传文档 |
| 需要稳定结构化输出 | 自然语言生成不稳定 | JSON schema、函数调用、后处理校验 |
| 需要长期记忆 | 模型无状态 | 数据库、用户画像系统 |
| 领域术语大量错误 | 预训练数据不覆盖 | 检索增强、少量样例、微调 |
| 输出质量难判断 | 主观任务缺少标准 | 建立评估集和评分标准 |
| 复杂多步骤任务 | 单次调用易出错 | Agent 工作流、思维链、工具调用 |
| 实时数据查询 | 模型知识有截止日期 | 函数调用、API 集成 |

## 常见错误与解决方案

### 错误 1：指令模糊

❌ **错误示例：**
```text
帮我优化一下这段文字
```

✅ **改进方案：**
```text
请优化以下产品描述，目标是提高转化率：

优化方向：
- 突出核心卖点（快速、易用、省钱）
- 用具体数据替代模糊描述
- 增加紧迫感（限时优惠）
- 语气更亲和，面向小企业主

输出：优化后的文字 + 简短说明（改了什么、为什么）
```

### 错误 2：一次要求太多

❌ **错误示例：**
```text
分析这份报告，总结要点，找出问题，给出建议，并预测未来趋势
```
**问题：** 任务太多，模型容易顾此失彼

✅ **改进方案：** 拆分成多个提示词，或明确优先级

```text
请先完成第一步：从报告中提取 5 个核心要点
[等模型输出后]
现在基于这些要点，分析存在的主要问题（不超过 3 个）
```

### 错误 3：没有提供必要背景

❌ **错误示例：**
```text
这个数据正常吗？
[贴一个数字]
```

✅ **改进方案：**
```text
背景：这是我们 SaaS 产品的月活数据
历史数据：过去 6 个月平均 MAU 为 12,000，波动范围 ±15%
本月数据：MAU = 9,500

问题：这个数据是否异常？如果是，可能的原因有哪些？
```

### 错误 4：期望模型有最新信息

❌ **错误示例：**
```text
告诉我今天的天气
```
**问题：** 模型训练数据有截止日期，无法获取实时信息

✅ **改进方案：**
- 使用支持联网搜索的模型
- 或通过函数调用集成天气 API
- 或明确告知："基于历史数据，XX月份的天气通常是..."

### 错误 5：未验证结构化输出

❌ **错误示例：**
```text
给我输出 JSON 格式的用户信息
```
然后直接把输出传给程序 → 解析失败

✅ **改进方案：**
1. 在提示词中给出完整 JSON schema
2. 要求模型"只输出 JSON，不要解释"
3. 程序端做 try-catch 和格式校验
4. 如果解析失败，重试或降级处理

## 入门练习（从易到难）

### 练习 1：总结文章
**任务：** 让模型把一篇 500 字文章总结成 5 条要点
**提示：** 明确"要点"是什么（核心观点？还是重要信息？），限定每条长度

### 练习 2：表格对比
**任务：** 让模型用表格比较 iPhone 和 Android 手机
**提示：** 指定对比维度（价格、生态、定制性等），要求客观

### 练习 3：错误诊断
**任务：** 给模型一段错误答案，要求它指出问题
**提示：** 提供正确答案作为参考，要求逐点对比

### 练习 4：风格模仿
**任务：** 提供一个示例产品描述，让模型按同样格式生成新的
**提示：** 明确哪些是"格式"（结构、长度），哪些是"内容"（具体信息）

### 练习 5：JSON 生成
**任务：** 要求模型输出 JSON，并用程序解析验证
**提示：** 给出完整 schema，要求"不要有其他文字"，用 try-catch 验证

### 练习 6：多步推理
**任务：** 让模型解一道数学应用题，要求显示步骤
**提示：** "先理解题意，再列式，最后计算"

### 练习 7：角色扮演
**任务：** 让模型扮演一位老师，解释一个复杂概念
**提示：** 明确学生背景（零基础？高中生？），要求用类比

## 常见问题（FAQ）

### Q1：提示词越长越好吗？
**A：** 不一定。长度要与任务复杂度匹配。简单任务（如"翻译这句话"）无需长篇大论；复杂任务需要详细指令。关键是"必要信息完整"而非"字数多"。

### Q2：为什么同样的提示词每次输出不同？
**A：** 模型的生成过程带有随机性（由 temperature 等参数控制）。如果需要稳定输出，可以：
- 降低 temperature（设为 0 或接近 0）
- 设置 seed 参数固定随机种子
- 对关键任务运行多次取共识

### Q3：提示词能让模型变聪明吗？
**A：** 不能。提示词只能"引导"模型发挥已有能力，无法让模型超越其训练数据和架构限制。如果任务需要模型不具备的知识或推理能力，需要考虑：
- 更换更强模型
- 提供外部知识（RAG）
- 微调模型

### Q4：如何判断提示词质量？
**A：** 好的提示词应该：
1. **稳定性**：多次运行，输出质量稳定（格式、准确度）
2. **准确性**：输出符合预期，错误率低
3. **高效性**：token 消耗合理，无冗余内容
4. **可维护性**：结构清晰，他人能看懂和修改

建议：建立 10-30 条评估样本，每次修改提示词后对比效果

### Q5：中文提示词和英文提示词有区别吗？
**A：** 对于多语言模型（如 GPT、Claude），理论上差异不大，但实践中：
- 英文训练数据更多，某些复杂推理任务英文可能略优
- 中文在特定领域（中国市场、文化语境）更准确
- 建议：用哪种语言就用哪种，如果效果不佳再尝试切换
