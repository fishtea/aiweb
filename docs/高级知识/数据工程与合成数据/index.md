# 数据工程与合成数据

模型能力很大程度上受数据质量影响。无论是预训练、微调、RAG 还是评估，数据工程都决定了系统上限，合成数据则是补齐稀缺场景、构造评估集和提升指令覆盖的重要手段。

## 数据在 AI 系统中的位置

| 场景 | 数据目标 |
|------|----------|
| 预训练 | 覆盖广泛知识、语言模式、代码和多模态内容 |
| 指令微调 | 教会模型遵循任务、格式、语气和安全边界 |
| 偏好优化 | 用人类或模型偏好样本塑造回答风格和对齐表现 |
| RAG | 提供可追溯、可更新、带权限的外部知识 |
| 评估 | 用代表性样例衡量质量、鲁棒性、安全性和回归风险 |

## 数据治理流程

1. 明确数据用途：训练、检索、评估、监控或反馈。
2. 定义数据 schema：字段、来源、时间、权限、标签和质量评分。
3. 清洗去重：删除重复、乱码、广告、页眉页脚、低质量模板文本。
4. 分层抽样：覆盖主任务、长尾任务、失败案例和高风险场景。
5. 标注与审核：建立标注指南、冲突处理和一致性检查。
6. 版本管理：记录数据集版本、构建脚本、过滤规则和变更说明。
7. 闭环迭代：把线上失败案例转化为评估集、训练集或知识库修订。

## 合成数据适用场景

- 真实样本稀缺，但任务规则明确。
- 需要覆盖边界条件、异常输入和安全对抗样例。
- 需要把专家知识转成问答、步骤、分类或工具调用样本。
- 需要扩展评估集，验证 Prompt、RAG 或 Agent 工作流是否回归。

## 合成数据风险

| 风险 | 处理方式 |
|------|----------|
| 幻觉扩散 | 合成前提供可靠来源，合成后做事实校验 |
| 分布偏移 | 混合真实样本，定期对比线上输入分布 |
| 格式单一 | 使用多种模板、难度、语气和错误类型 |
| 标签污染 | 分离训练集、验证集和测试集，避免同源泄漏 |
| 安全问题 | 对敏感、违法、隐私内容做过滤和审查 |

## 高质量样本特征

- 输入真实，包含业务上下文，而不是过度理想化问题。
- 输出可验证，最好有引用、规则、计算过程或人工标准答案。
- 难度分层，覆盖简单、普通、困难和拒答场景。
- 标签清晰，能说明为什么这个答案是好的或差的。
- 可追溯，知道样本来自真实用户、专家编写还是模型合成。

### 合成数据的进阶实践

2024-2025 年合成数据从"用 GPT 生成问答对"走向系统化方法论：

| 方法 | 说明 |
|------|------|
| 自我蒸馏（Self-Distillation） | 大模型生成数据蒸馏给小模型，保留能力降低成本 |
| 拒绝采样（Rejection Sampling） | 生成多候选，用奖励模型或规则筛选最优，提升数据质量 |
| 可验证奖励合成 | 对数学/代码等有标准答案的任务，只保留通过验证的样本 |
| 多样性驱动 | 用聚类确保合成样本覆盖不同难度和类型，避免模式坍缩 |
| 合成 + 人工混合 | 合成初稿 + 人工修订，平衡规模与质量 |

> 关键警示：纯模型合成数据反复训练会导致"模型坍缩"（model collapse）——分布尾部信息逐渐丢失。生产中必须混合真实数据，并对合成数据做去重和质量评分。
