# AI 应用部署运维

AI 应用上线后，运维关注点不仅是服务可用，还包括模型质量、推理成本、延迟、数据安全、提示词版本、检索索引和工具调用风险。

## 运维对象

| 对象 | 需要监控的内容 |
|------|----------------|
| 模型服务 | QPS、并发、首 token 延迟、总延迟、错误率、超时、成本 |
| RAG 链路 | 文档更新、索引构建、召回质量、引用覆盖率、空召回率 |
| Agent 工具 | 调用次数、参数错误、权限拒绝、外部 API 失败、重试次数 |
| Prompt 版本 | 模板变更、输出格式错误、质量回归和用户反馈 |
| 数据安全 | 敏感信息、越权访问、日志脱敏和审计记录 |

## 上线前清单

- 是否有评估集覆盖核心任务和高风险场景。
- 是否记录 Prompt、模型、参数、检索结果和工具调用版本。
- 是否设置超时、重试、限流、熔断和降级回答。
- 是否对用户输入和模型输出做必要的安全过滤。
- 是否能追踪一次回答从输入到最终输出的完整链路。
- 是否有成本预算、告警阈值和异常流量处理策略。

## 日志与追踪

生产日志不应只保存最终回答。建议至少记录：

- 请求 ID、用户或租户标识、入口和时间。
- 模型名称、参数、上下文长度、token 用量和耗时。
- 检索查询、召回文档 ID、重排序分数和引用来源。
- 工具调用名称、参数摘要、状态码、耗时和失败原因。
- 输出解析状态、拒答原因、用户反馈和人工审核结果。

涉及隐私或商业敏感内容时，应保存摘要、哈希或脱敏字段，而不是完整明文。

## 降级策略

| 故障 | 可选降级 |
|------|----------|
| 模型 API 超时 | 切换备用模型、缩短上下文、返回可解释的稍后重试 |
| 向量库不可用 | 使用关键词检索、缓存答案或静态 FAQ |
| 工具 API 失败 | 停止执行动作，返回失败原因和人工处理入口 |
| 输出格式错误 | 自动重试结构化修复，超过阈值进入人工队列 |
| 成本异常 | 降低并发、切小模型、关闭低优先级批任务 |

### 多级降级链路

生产系统应设计分层降级，避免单点故障直接暴露给用户：

```
主模型（高能力）
  → 超时/错误 → 备用模型（同类不同厂商）
  → 仍失败 → 语义缓存命中（历史相似问答）
  → 未命中 → 小模型快速兜底回复
  → 仍失败 → 静态 FAQ / 转人工
```

设计要点：

- **熔断器**：某下游连续失败达到阈值后短路，避免级联雪崩。
- **超时分级**：首 token 超时（影响体感）与总时长超时（影响完整性）分开设置。
- **优雅降级回复**：降级时告知用户"当前服务降级，结果可能不完整"，而非静默返回低质量答案。
- **可观测**：每次降级都记录触发原因，用于事后归因和容量规划。

## 质量闭环

1. 收集线上失败案例和用户负反馈。
2. 标注失败类型：检索失败、幻觉、格式错误、工具错误、权限错误或需求不清。
3. 将高频问题加入离线评估集。
4. 修改 Prompt、索引、工具 schema、模型路由或业务规则。
5. 回归测试通过后灰度发布，并继续监控关键指标。
