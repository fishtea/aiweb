# AI 应用部署运维

AI 应用上线后，运维关注点不仅是服务可用，还包括模型质量、推理成本、延迟、数据安全、提示词版本、检索索引和工具调用风险。

## 运维对象

| 对象 | 需要监控的内容 |
|------|----------------|
| 模型服务 | QPS、并发、首 token 延迟、总延迟、错误率、超时、成本 |
| RAG 链路 | 文档更新、索引构建、召回质量、引用覆盖率、空召回率 |
| Agent 工具 | 调用次数、参数错误、权限拒绝、外部 API 失败、重试次数 |
| Prompt 版本 | 模板变更、输出格式错误、质量回归和用户反馈 |
| 数据安全 | 敏感信息、越权访问、日志脱敏和审计记录 |

## 上线前清单

- 是否有评估集覆盖核心任务和高风险场景。
- 是否记录 Prompt、模型、参数、检索结果和工具调用版本。
- 是否设置超时、重试、限流、熔断和降级回答。
- 是否对用户输入和模型输出做必要的安全过滤。
- 是否能追踪一次回答从输入到最终输出的完整链路。
- 是否有成本预算、告警阈值和异常流量处理策略。

## 日志与追踪

生产日志不应只保存最终回答。建议至少记录：

- 请求 ID、用户或租户标识、入口和时间。
- 模型名称、参数、上下文长度、token 用量和耗时。
- 检索查询、召回文档 ID、重排序分数和引用来源。
- 工具调用名称、参数摘要、状态码、耗时和失败原因。
- 输出解析状态、拒答原因、用户反馈和人工审核结果。

涉及隐私或商业敏感内容时，应保存摘要、哈希或脱敏字段，而不是完整明文。

## 降级策略

| 故障 | 可选降级 |
|------|----------|
| 模型 API 超时 | 切换备用模型、缩短上下文、返回可解释的稍后重试 |
| 向量库不可用 | 使用关键词检索、缓存答案或静态 FAQ |
| 工具 API 失败 | 停止执行动作，返回失败原因和人工处理入口 |
| 输出格式错误 | 自动重试结构化修复，超过阈值进入人工队列 |
| 成本异常 | 降低并发、切小模型、关闭低优先级批任务 |

### 多级降级链路

生产系统应设计分层降级，避免单点故障直接暴露给用户：

```
主模型（高能力）
  → 超时/错误 → 备用模型（同类不同厂商）
  → 仍失败 → 语义缓存命中（历史相似问答）
  → 未命中 → 小模型快速兜底回复
  → 仍失败 → 静态 FAQ / 转人工
```

设计要点：

- **熔断器**：某下游连续失败达到阈值后短路，避免级联雪崩。
- **超时分级**：首 token 超时（影响体感）与总时长超时（影响完整性）分开设置。
- **优雅降级回复**：降级时告知用户"当前服务降级，结果可能不完整"，而非静默返回低质量答案。
- **可观测**：每次降级都记录触发原因，用于事后归因和容量规划。

## 质量闭环

1. 收集线上失败案例和用户负反馈。
2. 标注失败类型：检索失败、幻觉、格式错误、工具错误、权限错误或需求不清。
3. 将高频问题加入离线评估集。
4. 修改 Prompt、索引、工具 schema、模型路由或业务规则。
5. 回归测试通过后灰度发布，并继续监控关键指标。

## 2026年LLM推理优化实战指南

### 核心瓶颈：内存带宽，而非计算

LLM 推理（尤其是自回归 token 生成阶段）的主要瓶颈是**内存带宽**（Memory-bandwidth bound），而不是算力。一块 H100 GPU 拥有 3.35 TB/s 内存带宽和 989 TFLOPS FP16 算力，但在解码阶段实际只用到 10-20% 的算力——GPU 大部分时间在等待数据传输。优化推理的核心思路是**减少内存中移动的数据量**，而非单纯增加 GPU。

### 关键优化技术

#### 1. KV Cache 优化

KV Cache 是推理优化中最重要的概念。对于 Llama 3 70B（80 层、8 个 KV 头（GQA）、头维度 128、FP16），单个 4096-token 序列的 KV Cache 约为 1.3 GB。batch size 为 32 时，KV Cache 膨胀到约 **41 GB**——往往超过量化后的模型权重本身。

**优化策略：**

- **PagedAttention（vLLM）**：以虚拟内存分页方式管理 KV Cache 块，将内存浪费从 60-80% 降至几乎为零，同一 GPU 可承载 **2-3 倍**的并发请求。已被主流推理框架广泛采用。
- **Prefix Caching（前缀缓存）**：对于系统提示词、少样本示例等公共前缀，KV Cache 只计算一次并复用。2000-token 的公共前缀向 1000 个用户提供时，可节省约 20 亿 token 的冗余计算。RAG 场景收益尤为显著。
- **GQA（分组查询注意力）**：Llama 3 在 64 个查询头间共享 8 个 KV 头——KV Cache 大小降低 **8 倍**（架构层面优化，非运行时）。

#### 2. 量化：用更少内存做更多事

| 格式 | 位宽 | 内存缩减 | 典型质量损失 | 最佳场景 |
|------|------|---------|------------|---------|
| FP8 (E4M3) | 8 | 2× | 可忽略 | H100/H200 生产推理 |
| INT8 (W8A8) | 8 | 2× | <1% | 通用生产推理 |
| INT4 (AWQ) | 4 | 4× | 1-2% | 内存受限 GPU，质量优于 GPTQ |
| INT4 (GPTQ) | 4 | 4× | 1-3% | 传统 INT4 方案 |
| GGUF (llama.cpp) | 2-8 | 2-8× | 随级别变化 | 本地/CPU 推理 |

> **经验法则**：H100/H200 首选 FP8，A100 首选 INT8 (AWQ)——内存减半，质量几乎无损失。仅在需要适配更小 GPU 才降到 INT4。低于 INT4 时复杂推理会有明显质量下降。2026 年新部署推荐 AWQ 作为默认 INT4 方案。

#### 3. Flash Attention：内存高效注意力机制

标准自注意力需计算 N×N 的注意力矩阵（O(N²) 内存），Flash Attention 将注意力操作融合为单个 CUDA kernel，在高速 SRAM 中分块流式计算，**内存复杂度从 O(N²) 降至 O(N)**——长上下文模型的关键支撑。目前已是所有主流框架的默认组件。如果长序列推理时出现 OOM，缺少 Flash Attention 往往是根因。

#### 4. 推测解码（Speculative Decoding）

用小模型（1-7B）快速"猜测" K 个候选 token（K=3-8），大模型在一个前向传播中并行验证所有候选。准确率高时（如代码生成），70-90% 候选被接受——每次大模型推理可产出 **3-6 个 token**，数学上保证输出分布不变。

- **适用场景**：延迟敏感、低并发的交互式场景（编程助手、实时 Agent）。
- **注意**：可能降低吞吐量（当 GPU 利用率已很高时），因为它减少了大模型的 batch 效率。

#### 5. 连续批处理（Continuous Batching）

静态批处理需等待最长请求完成才能插入新请求；连续批处理在任意请求完成后立即动态插入新序列，以 **token 级别**调度。GPU 利用率从 30-40% 跃升至 80-90%，显著降低单 token 成本。

### 2026年本地部署（On-Premise）

本地部署指在自有硬件上运行完整推理栈（模型权重 + 推理引擎 + API），无第三方 API 依赖。到 2025 年，约 71% 的 AI 基础设施已不在公有云上。

**决策四维度：**

- **利用率**：GPU 日均使用 > 4-5 小时、多年周期 → 本地部署。全利用率下，自有集群的单 token 成本比云按需定价低 **8-18 倍**。
- **体量**：< 1000 万 token/月（中等模型） → 云；大流量持续推理 → 本地。
- **数据主权**：GDPR 第 46 条、EU AI Act 最高罚款 €3500 万或全球营业额 7%——合规可强制本地部署。
- **延迟**：对实时性敏感的 Agent 场景，本地 GPU 可消除网络延迟抖动。

**模型选择要点：**

- **许可证优先**：Apache 2.0/MIT 覆盖 Qwen3、Mistral Small 3.x、DeepSeek V3/R1、Phi-4。Llama 4 需额外许可且非 OSI 认证。Gemma 需接受使用条款。
- **MoE vs 密集模型**：MoE（如 DeepSeek V3.2 671B/37B active）需将所有专家权重加载到 VRAM，往往需要多 GPU/多节点。密集模型延迟可预测，更适合单节点部署。
- **2026 年推荐的本地部署模型**：Qwen3.6-27B（通用）、Gemma 4 31B（通用/RAG）、Qwen3-Coder-Next 80B（编程）、Devstral 2 123B（密集、延迟可预测）、gpt-oss-20b（边缘/CPU）。

**推理引擎选型：** vLLM 是 2026 年最广泛采用的本地推理引擎（PagedAttention 标配），NVIDIA TensorRT-LLM 追求极致性能，SGLang 在结构化输出上有优势。

### 2026年推理成本管理与弹性伸缩实战

#### 云 vs 本地：2026年成本对照

根据行业部署数据，本地 GPU 集群在全利用率下比云按需定价便宜 **8-18 倍**，但前提是 GPU 日均使用 > 4-5 小时：

| 部署方式 | 月成本（70B 模型/中等流量） | 适用场景 |
|---------|--------------------------|---------|
| 云 API（GPT-4o/Claude） | $10K-50K+ | 低延迟启动、不固定流量、PoC |
| 云 GPU 按需（H100×2） | $5K-10K | 中等流量、弹性需求 |
| 云 GPU 预留（1年） | $3K-6K | 稳定流量、成本可控 |
| 自建 H100×2 集群 | $2K-4K | 高利用率、数据主权 |
| 自建消费级（3090/4090×4） | $1K-2K | 小模型推理、原型 |

> **2026 年关键变化**：NVIDIA H200 / B200 上市后，H100 云实例价格下降 20-30%。对于 < 1000 万 token/月的低流量场景，云 API 仍是最经济的零运维选择。

#### GPU 利用率优化

GPU 利用率是推理成本的直接杠杆——每提升 10% 利用率≈成本降 10%：

- **Continuous Batching**：利用率从 30-40% → 80-90%（vLLM 默认启用）
- **Prefix Caching**：共享 System Prompt 场景下，TTFT 降低 50-80%，等效节省 GPU 时间
- **量化**：FP8（H100）内存减半→同 GPU 承载 2× 并发请求
- **Speculative Decoding**：低并发延迟敏感场景 2-3× 解码加速

**实际案例**：一台 H100 运行 Llama 3 70B FP8 + vLLM Continuous Batching，典型并发 32 时月成本约 $1,500-2,500（自建摊销），同等能力云 API 约 $8,000-15,000。

#### 弹性伸缩模式

```
                          ┌──────────────┐
                          │  Load Balancer│
                          │  (Nginx/Envoy)│
                          └──────┬───────┘
                                 │
              ┌──────────────────┼──────────────────┐
              ▼                  ▼                  ▼
    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
    │  Base Pool  │    │  Burst Pool │    │  Fallback   │
    │  (常驻 2×H100)│   │ (竞价 GPU)  │    │  (小模型)    │
    └─────────────┘    └─────────────┘    └─────────────┘
```

- **Base Pool（常驻池）**：处理基线流量，预留实例或自建。运行主模型（如 Llama 3 70B）。
- **Burst Pool（弹性池）**：应对流量尖峰，使用竞价/Spot 实例（成本降低 60-70%）。由 KEDA/Prometheus 触发扩容。
- **Fallback（降级链路）**：竞价实例不可用时，切换至小模型（如 Llama 3 8B）+ 缓存兜底。

#### vLLM 生产监控指标

vLLM 通过 `--enable-metrics` 暴露 Prometheus 端点，关键指标：

| 指标 | 含义 | 告警阈值建议 |
|------|------|------------|
| `vllm:request_success_total` | 成功请求数 | —（趋势监控） |
| `vllm:time_to_first_token_seconds` | 首 token 延迟 | P95 > 2s 告警 |
| `vllm:time_per_output_token_seconds` | 每 token 生成延迟 | P95 > 50ms 告警 |
| `vllm:num_requests_running` | 当前并发请求数 | > max_num_seqs × 0.8 |
| `vllm:num_requests_waiting` | 排队请求数 | > 0 持续 30s |
| `vllm:gpu_cache_usage_perc` | KV 缓存使用率 | > 90% 告警（即将 OOM） |
| `vllm:prefix_cache_hit_rate` | 前缀缓存命中率 | < 30%（优化信号） |

**Grafana Dashboard 推荐布局：**

1. **延迟面板**：TTFT P50/P95/P99、TPOT（每 token 延迟）
2. **吞吐面板**：请求数/秒、token/秒（输入+输出）
3. **资源面板**：GPU 利用率、GPU 显存、KV 缓存使用率
4. **质量面板**：错误率、超时率、缓存命中率
