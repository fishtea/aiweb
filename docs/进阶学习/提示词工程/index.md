# 提示词工程

> 提示词工程（Prompt Engineering）是设计和优化输入提示词以引导 AI 模型产生期望输出的方法论。它是当前与 LLM 交互最核心的技能之一。

## 🤔 什么是提示词工程？

提示词工程不是简单的"提问技巧"，而是一套系统化的方法论：

- **设计**：构造清晰、结构化的输入
- **优化**：迭代改进提示词以提升输出质量
- **评估**：量化比较不同提示词的效果
- **管理**：版本控制、模板化管理

### 为什么重要？

| 原因 | 说明 |
|------|------|
| **直接决定输出质量** | 好的提示词能让模型能力提升数倍 |
| **节省成本** | 有效的提示词减少 token 消耗和重试次数 |
| **可控性** | 引导模型行为，减少随机性和有害输出 |
| **可复现** | 标准化的提示词确保一致的结果 |

## 📋 提示词基本结构

LLM 的聊天接口通常包含三种角色消息：

### System Message（系统消息）

设定模型的行为、角色和边界：

```
你是一个资深的 Python 开发者，擅长编写高质量、可维护的代码。
回答要简洁、准确，包含必要的代码示例。
```

### User Message（用户消息）

用户的输入或问题。可以包含：

- **指令**：明确告诉模型做什么
- **上下文**：提供背景信息
- **示例**：展示期望的输出格式
- **约束**：设定限制条件

### Assistant Message（助手消息）

模型的回复。在多轮对话中，之前的回复也是上下文的一部分。

### 完整的提示词结构

```python
messages = [
    {"role": "system", "content": "你是代码审查专家。"},
    {"role": "user", "content": "请审查以下代码：\n```python\ndef calc(a,b):\n    return a/b\n```"},
    {"role": "assistant", "content": "需要注意除零错误..."}
]
```

## 🎯 核心技术

### 1. Few-Shot（少样本学习）

提供几个输入输出示例，让模型"照猫画虎"：

```
将以下中文句子翻译为英文：

例子1：
中文：今天天气真好
英文：The weather is nice today

例子2：
中文：这家餐厅的菜很好吃
英文：The food at this restaurant is delicious

现在请翻译：
中文：这个电影让我很感动
英文：
```

### 2. Chain-of-Thought（思维链）

引导模型进行逐步推理，特别适合数学、逻辑问题：

```
问题：商店有 15 个苹果，卖了 7 个，又进了 10 个，现在有多少个？

逐步推理：
1. 一开始有 15 个苹果
2. 卖了 7 个后：15 - 7 = 8 个
3. 又进了 10 个后：8 + 10 = 18 个

答案：18 个
```

### 3. Chain-of-Density（密度链）

逐步要求模型在保持信息完整的前提下让内容更精炼：

```
第一轮：请用 3 句话概括这篇文章
第二轮：请将上述概括压缩为 2 句话，保留所有关键信息
第三轮：请用 1 句话包含所有核心要点
```

### 4. Self-Consistency（自洽性）

多次采样后投票选出最一致的回答，能显著提升推理准确率。

## 🎨 高级模式

### 角色扮演（Persona）

为模型设定具体的角色身份：

```
你是一位拥有 20 年经验的量子物理学家。
请用通俗易懂的语言向高中生解释量子纠缠。
避免使用过于专业的术语。
```

### 格式控制

精确控制输出格式，便于程序化处理：

```
请分析以下用户评论，以 JSON 格式输出：
{
  "情感": "正面/负面/中性",
  "关键词": ["...", "..."],
  "评分": 1-5,
  "摘要": "一句话总结"
}
```

### 负面提示（Negative Prompts）

告诉模型不要做什么：

```
请写一篇产品介绍。
要求：
- 不要使用营销夸张用语
- 不要使用"革命性""颠覆性"等词汇
- 不要超过 200 字
- 不要包含价格信息
```

### 思维树（Tree-of-Thoughts）

让模型同时探索多个思路：

```
关于这个问题的解决方案，请提供 3 种不同的思路。
对于每种思路，分析其优缺点。
然后综合评估，给出推荐方案。
```

## 📁 提示词模板与版本管理

### 模板化

将提示词中的变量分离出来，便于复用：

```python
prompt_template = """
你是一个{role}专家。
请针对以下{input_type}进行分析：
{input_content}

要求：
1. 从{aspects}角度分析
2. 输出格式：{output_format}
"""
```

### 版本管理

使用 Git 或专门的 Prompt 管理工具：

- **版本记录**：每次修改记录变更原因
- **A/B 测试**：对比不同版本的效果
- **标签管理**：按用途（翻译、代码、分析）分类
- **回滚能力**：效果下降时快速回退

## 📊 提示词评估

### 评估维度

| 维度 | 指标 | 评估方法 |
|------|------|---------|
| **准确性** | 是否正确回答 | 人工评分、自动比对 |
| **相关性** | 是否切题 | 文本相似度 |
| **完整性** | 是否全面 | 覆盖度检查 |
| **格式** | 是否符合要求 | 格式校验 |
| **效率** | 消耗的 token 数 | 统计计算 |

### 评估方法

1. **人工评估**：通过评分员打分（最准确但最慢）
2. **LLM 评估**：用 GPT-4 等模型评估输出质量
3. **自动评估**：用规则、正则、测试用例验证
4. **A/B 测试**：线上流量对比

## ⚠️ 常见错误

| 错误 | 表现 | 解决方案 |
|------|------|---------|
| **指令模糊** | 模型输出不符合预期 | 使用具体、明确的指令 |
| **过度约束** | 模型变得机械僵硬 | 适当放松约束，保留灵活性 |
| **忽略上下文** | 模型"忘记"之前的对话 | 将关键信息放入系统消息 |
| **否定表述** | "不要想熊猫"反而会想熊猫 | 使用肯定表述代替否定 |
| **链式错误** | 多步推理中某一步错了 | 让模型在每个步骤后验证 |

## 💡 实践建议

1. **系统消息优先**：把最重要的指令放在 System Message 中
2. **指令前置**：将核心指令放在提示词开头
3. **少即是多**：简洁明确的提示通常比冗长的提示效果更好
4. **迭代优化**：没有完美的第一次提示词，持续迭代是常态
5. **记录测试用例**：维护一组标准测试用例，每次修改后都验证
