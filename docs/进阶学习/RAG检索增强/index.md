# RAG 检索增强生成

> 让语言模型不再局限于训练数据，通过连接外部知识库实现实时、准确的信息检索与生成。RAG 是目前落地最广泛的 LLM 应用架构。

## 🤔 为什么需要 RAG？

大语言模型的知识存在三个核心局限：

| 局限 | 表现 | RAG 的解决方案 |
|------|------|---------------|
| **知识截止** | 模型不知道最新事件 | 实时从外部数据库检索最新信息 |
| **幻觉** | 模型自信地编造事实 | 基于检索到的真实文档生成答案 |
| **领域知识欠缺** | 缺乏私有/专业知识 | 连接企业知识库或个人文档 |

> **RAG vs 微调**：RAG 适合需要动态知识的场景，微调适合需要固化和改变模型行为与风格的场景。两者可以互补使用。

## 🏗️ RAG 架构

RAG 的核心流程分为三个阶段：

```
用户查询
   │
   ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Retriever │ →  │Augmenter │ →  │Generator │
│ (检索器)   │    │(增强器)   │    │(生成器)   │
└──────────┘    └──────────┘    └──────────┘
   │               │               │
   ▼               ▼               ▼
检索相关文档    拼接上下文      生成最终答案
```

### 1️⃣ 检索器（Retriever）

将用户查询转换为向量，在文档库中找到最相关的内容。

### 2️⃣ 增强器（Augmenter）

将检索到的文档与原始查询组合成提示词（Prompt）。

典型模板：
```
基于以下资料回答问题：
[检索到的文档 1]
[检索到的文档 2]
...
问题：[用户查询]
```

### 3️⃣ 生成器（Generator）

将增强后的提示词输入 LLM，生成基于事实的答案。

## 📄 文档分块策略

文档分块是 RAG 效果的基石。分块过大或过小都会影响检索质量。

### 分块方法

| 方法 | 描述 | 适用场景 |
|------|------|---------|
| **固定长度分块** | 按字符/token 数切分 | 通用场景 |
| **语义分块** | 按段落、句子边界切分 | 文章、报告 |
| **递归分块** | 递归缩小分块大小 | 复杂文档 |
| **特定格式分块** | 按 Markdown 标题、代码函数切分 | 结构化文档 |

### 分块参数

| 参数 | 建议值 | 说明 |
|------|--------|------|
| 块大小 | 256-1024 tokens | 太小缺少上下文，太大降低检索精度 |
| 重叠大小 | 10%-20% 块大小 | 保留上下文边界信息 |
| 嵌入模型 | text-embedding-3-small, bge-large | 选择高质量的嵌入模型 |

## 🔤 嵌入模型与向量数据库

### 嵌入模型

将文本转换为向量表示。常用的嵌入模型：

- **OpenAI**: text-embedding-3-small / 3-large（闭源）
- **BGE**: BAAI/bge-large-zh-v1.5（中文推荐）
- **E5**: intfloat/e5-mistral-7b-instruct
- **GTE**: Alibaba-NLP/gte-Qwen2

### 向量数据库

存储和检索向量的专用数据库：

| 数据库 | 类型 | 特点 |
|--------|------|------|
| **Chroma** | 轻量级 | 本地运行，适合原型开发 |
| **Qdrant** | 高性能 | 支持过滤、分组，生产级 |
| **Pinecone** | 云服务 | 全托管，无需运维 |
| **Milvus** | 分布式 | 超大规模场景 |
| **pgvector** | PostgreSQL 扩展 | 与关系数据库结合 |

## 🔍 检索方法

### 稠密检索（Dense Retrieval）

使用嵌入模型将文本转为向量，通过余弦相似度或内积计算相关性。

**优点**：理解语义，不受词汇差异影响
**缺点**：需要向量数据库，对罕见词处理一般

### 稀疏检索（Sparse Retrieval）

基于 BM25 等传统关键词匹配方法。

**优点**：精确匹配，无需训练，处理罕见词好
**缺点**：无法理解语义，同义词匹配差

### 混合检索（Hybrid Retrieval）

结合稠密和稀疏检索，取长补短：

```
score = α × dense_score + (1-α) × sparse_score
```

α 通常设置为 0.5-0.7，可根据具体场景调优。

## 📊 RAG 评估

### 评估维度

| 维度 | 指标 | 含义 |
|------|------|------|
| **忠实度** | Faithfulness | 生成内容是否基于检索文档 |
| **相关性** | Relevance | 检索文档和问题是否相关 |
| **完整性** | Completeness | 答案是否覆盖所有必要信息 |
| **上下文精度** | Context Precision | 检索结果中相关文档的比例 |

### 评估工具

- **RAGAS**：专为 RAG 设计的多维度评估框架
- **TruLens**：提供 RAG 三元组评估
- **LangSmith**：LangChain 生态的追踪和评估平台

## 🚀 高级 RAG 技术

### Self-RAG

让 LLM 自我反思检索结果：
1. 检索文档 → 判断是否需要检索
2. 生成答案 → 评估答案是否基于文档
3. 自我纠错 → 标记不确定的部分

### Agentic RAG

将 RAG 流程交给 Agent 自主决策：
- Agent 决定何时检索、检索什么
- 支持多步检索和次优结果修正
- 可调用多个工具（搜索、数据库、API）

### RAPTOR

递归摘要式检索：
1. 对文档分块并生成摘要
2. 对摘要再次生成更高层摘要
3. 构建文档摘要树
4. 检索时匹配不同粒度的内容

### Graph RAG

基于知识图谱的 RAG：
- 从文档中提取实体和关系
- 构建知识图谱
- 通过图遍历进行多跳推理
- 微软开源，适合全局性问题

## 💡 实践建议

1. **先看检索效果**：单独评估检索器，确保召回率达标
2. **块大小实验**：不同文档类型需要不同的分块策略
3. **嵌入模型匹配**：中文场景务必使用中文优化的嵌入模型
4. **元数据过滤**：使用文档元数据（日期、来源、类型）过滤不相关结果
5. **多路召回**：混合使用多种检索方法

## 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 检索不到相关内容 | 块太小或嵌入模型不匹配 | 增加块大小，换用更好的嵌入模型 |
| 生成内容脱离文档 | Prompt 设计不当 | 在 Prompt 中强调"仅基于检索内容回答" |
| 检索结果太多 | 相似度阈值太低 | 提高阈值，限制返回文档数 |
