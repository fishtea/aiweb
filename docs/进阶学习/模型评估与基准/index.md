# 模型评估与基准

> 没有测量就没有改进。模型评估是 AI 开发中最容易被低估但至关重要的环节——它决定了你能否科学地比较模型、发现弱点、跟踪进展。

## 🤔 为什么要评估模型？

| 目的 | 说明 |
|------|------|
| **比较模型** | 在多个候选模型中选出最适合的 |
| **发现弱点** | 识别模型在哪些任务上表现差 |
| **跟踪进展** | 量化微调/优化前后的效果提升 |
| **风险控制** | 发现安全漏洞和有害输出 |
| **成本决策** | 平衡效果与成本，选择最优方案 |

## 📚 主流 NLP 基准

### 知识理解

| 基准 | 说明 | 评估能力 |
|------|------|---------|
| **MMLU** | 57 个学科的多选题 | 世界知识、专业理解 |
| **MMLU-Pro** | MMLU 增强版，更难 | 更深层次的知识推理 |
| **C-Eval** | 中文综合评测 | 中文领域知识 |

MMLU 是 LLM 评估的首选基准，覆盖从高中到专业水平的各学科知识。

### 推理能力

| 基准 | 说明 | 评估能力 |
|------|------|---------|
| **HellaSwag** | 选择最合理的结尾 | 常识推理 |
| **GSM8K** | 小学数学应用题 | 数学推理 |
| **MATH** | 竞赛级数学题 | 高级数学推理 |
| **ARC** | 科学问答 | 科学推理 |

### 代码能力

| 基准 | 说明 | 评估能力 |
|------|------|---------|
| **HumanEval** | 编写函数通过测试 | 代码生成 |
| **MBPP** | 基础 Python 编程 | 基础编码 |
| **SWE-bench** | 真实 GitHub Issue | 完整软件工程 |
| **BigCodeBench** | 综合编程任务 | 实用编程能力 |

### 安全与对齐

| 基准 | 说明 |
|------|------|
| **TruthfulQA** | 检测模型是否倾向于生成常见误解 |
| **BBQ** | 检测社会偏见 |
| **HarmBench** | 检测有害输出 |

## 🔬 LLM 特定评估

### Chatbot Arena

由 LMSYS 组织的竞技场式评估：

- 用户匿名与两个模型对话
- 用户投票选出更好的回答
- 使用 Elo 评分系统排名
- 目前最接近真实使用场景的评估

### MT-Bench

多轮对话评估：

- 覆盖 8 个类别：写作、角色扮演、推理、数学、代码等
- 每轮包含多轮对话
- 使用 GPT-4 作为评判者评分
- 生成 1-10 分的评分

### AlpacaEval

自动评估框架：

- 从 805 个测试问题池中采样
- 使用 GPT-4 / Claude 作为裁判
- 计算与参考模型（如 GPT-4）的胜率
- 速度快，成本低

## 📊 评估维度

### 六维评估框架

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│  知识    │  │  推理    │  │  编码    │
│  广度    │  │  能力    │  │  能力    │
└──────────┘  └──────────┘  └──────────┘
┌──────────┐  ┌──────────┐  ┌──────────┐
│  安全    │  │  对齐    │  │  效率    │
│  性      │  │  程度    │  │  成本    │
└──────────┘  └──────────┘  └──────────┘
```

### 关键评估指标

| 指标 | 含义 | 评估方法 |
|------|------|---------|
| **准确率** | 正确回答的比例 | 自动比对答案 |
| **F1 分数** | 精确率和召回率的调和平均 | 生成任务 |
| **困惑度** | 模型对文本的预测能力 | 语言建模 |
| **胜率** | 与参考模型的对比胜率 | Elo / 胜率统计 |
| **延迟** | 生成首个 token 的时间 | 性能测试 |
| **吞吐量** | 每秒生成的 token 数 | 性能测试 |

## 🤖 自动评估 vs 人工评估

### 自动评估

**优点**：
- 速度快、成本低
- 可重复、标准化
- 适合大规模评估

**缺点**：
- 无法评估创造性、流畅性等主观维度
- 裁判模型可能有偏见

### 人工评估

**优点**：
- 最准确的评估方式
- 能捕捉细微差异
- 是"黄金标准"

**缺点**：
- 成本高、速度慢
- 不同评分员一致性问题
- 难以规模化

**最佳实践**：自动评估做快速筛选，人工评估做最终决断。

## 🛠️ 运行评估

### lm-eval-harness

最流行的 LLM 评估框架，由 EleutherAI 开发：

```bash
# 安装
pip install lm-eval

# 评估 MMLU（零样本）
lm_eval --model hf \
  --model_args pretrained=Qwen/Qwen2.5-7B \
  --tasks mmlu \
  --num_fewshot 5

# 评估多个基准
lm_eval --model hf \
  --model_args pretrained=Qwen/Qwen2.5-7B \
  --tasks mmlu,gsm8k,hellaswag,human_eval \
  --batch_size auto
```

### 自定义评估

针对你自己的数据集评估：

```python
from lm_eval import simple_evaluate

results = simple_evaluate(
    model="hf",
    model_args="pretrained=Qwen/Qwen2.5-7B",
    tasks=["custom_task"],
    # 在 lm_eval/tasks/ 下注册自定义任务
)
```

## 📈 如何解读结果

### 不要只看一个数字

```json
{
  "MMLU": 0.72,
  "GSM8K": 0.84,
  "HumanEval": 0.65,
  "HellaSwag": 0.78
}
```

**解读**：
- MMLU 0.72：扎实的基础知识
- GSM8K 0.84：数学推理能力强
- HumanEval 0.65：代码能力有待提升
- **结论**：这个模型适合需要数学推理的应用，但不适合编程任务

### 评估的常见陷阱

| 陷阱 | 表现 | 避免方法 |
|------|------|---------|
| **数据污染** | 模型可能见过测试集 | 使用时间戳分割，选择新基准 |
| **过拟合基准** | 过度优化特定基准 | 使用多样化评估 |
| **单次评估** | 结果受随机性影响 | 多次运行取平均 |
| **忽略标准差** | 不同 seed 结果差异大 | 报告均值和置信区间 |
| **选择偏差** | 只报告最好的结果 | 报告所有结果 |

## 💡 实践建议

1. **建立基线**：评估前先确定参考模型和基线分数
2. **多样性优先**：覆盖多个评估维度，不要只依赖一个基准
3. **持续评估**：将评估集成到开发流程中（CI/CD）
4. **公开透明**：记录评估设置（seed、few-shot 数、温度参数）
5. **结合实际**：基准分数好不意味着实际效果好，补充真实场景测试
