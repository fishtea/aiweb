# 内容治理与采集规划

本项目的目标不是把搜索结果不断堆进文章，而是把 AI 学习资料沉淀成“有路径、有正文、有资源、有更新记录”的知识库。

## 核心原则

| 原则 | 说明 |
|------|------|
| 正文稳定 | 教程正文用于解释概念、机制、实践步骤和误区，默认由人工维护 |
| 资源动态 | 自动采集内容只进入后台资源库和更新报告，不作为站外链接列表发布 |
| 路径优先 | 初级、进阶、高级、模型、工具、Agent 实践要形成递进关系 |
| 来源分级 | 优先官方文档、论文、课程、主流开源项目和高质量技术文章 |
| 可追溯 | 每条资源保留 URL、来源域名、首次采集日期、最近出现日期和质量分 |
| 可重建 | 页面中的资料整理状态可从 `data/resources.json` 重新生成 |

## 内容分层

```text
首页
├── 推荐学习路径：告诉读者先学什么、后学什么
├── 分类入口：初级、进阶、高级、模型、工具、Agent
└── 最新更新：站点级变化和每日采集报告入口

专题页
├── 教程正文：人工维护，保持循序渐进
├── 实践建议：最小示例、验证方法、常见误区
└── 资料整理状态：脚本自动生成，只显示候选资料状态，不发布原始搜索结果

资源库
├── data/resources.json：结构化资源源数据
└── docs/知识库/更新报告/YYYY-MM-DD.md：每日采集摘要
```

## 采集流水线

```text
统一分类配置
  ↓
搜索关键词生成
  ↓
多来源搜索结果抓取
  ↓
URL 归一化、跨来源去重、屏蔽域名过滤
  ↓
URL 可访问性验证
  ↓
来源域名与关键词质量评分
  ↓
标题/摘要翻译
  ↓
写入 data/resources.json
  ↓
生成每日更新报告
  ↓
重建专题页资料整理状态区块
  ↓
VitePress 构建与部署
```

## 资源质量策略

优先采集：

- 官方文档：OpenAI、Anthropic、Google、Hugging Face、PyTorch、TensorFlow、LangChain、LlamaIndex、vLLM 等。
- 论文和技术报告：arXiv、模型官方技术报告、框架白皮书。
- 系统课程：DeepLearning.AI、Coursera、Microsoft Learn、Google ML Crash Course。
- 高质量开源项目：维护活跃、README 完整、示例清晰的 GitHub 仓库。
- 工程实践文章：有代码、有架构图、有边界条件、有评估方法。

降低优先级或屏蔽：

- 纯营销页、内容农场、无来源转载。
- 只有排行榜但无评测方法的文章。
- 过时 API 教程。
- 社交媒体短内容和不可稳定访问的页面。

## 多来源采集策略

采集脚本默认支持多个信息源，并把不同来源统一成同一种资源结构，避免每个页面直接依赖某个搜索 API。

| 信息源 | 用途 | 配置 |
|--------|------|------|
| Tavily | Agent/RAG 友好的搜索结果，带摘要、URL、相关度 | `TAVILY_API_KEY` |
| ddgs | 无 API key 的免费 fallback | 默认可用 |
| GitHub API | 热门 AI 项目、stars、forks、活跃度等结构化数据 | 可选 `GITHUB_TOKEN` |

默认信息源顺序为 `tavily,ddgs,github`。可以通过命令行或环境变量调整：

```bash
python scripts/collect.py --providers tavily,ddgs,github
AIWEB_SEARCH_PROVIDERS=tavily,ddgs python scripts/collect.py
```

去重与验证规则：

- 先对 URL 去除跟踪参数、统一域名大小写和路径结尾斜杠。
- 再使用 `dedupe_key` 按域名 + 路径做跨来源去重，避免同一资源被 Tavily、ddgs、GitHub API 重复写入。
- 默认会请求候选 URL，只有可访问资源才会进入 `data/resources.json`。
- 调试、无网络或只验证逻辑时可使用 `--skip-url-verify`。
- 同一资源被多个来源命中时，保留 `providers` 列表、更新 `seen_count` 和最高质量分。

翻译规则：

- 英文标题和摘要会写入 `title_cn`、`summary_cn`。
- 中文占比足够高的内容不重复翻译。
- 翻译失败不会阻塞采集，保留英文原文并在日志中输出警告。

内容成长规则：

- 自动采集只更新后台资源库和页面“资料整理状态”，不直接发布原始链接，不直接改写正文。
- 每周阅读新增候选资料，把高价值内容归纳成中文正文，吸收到教程、模型页、工具页或 Agent 实战页。
- 涉及模型版本、价格、API 额度、排行榜、star 数的数据必须标注获取日期。
- 资源库只保留可验证来源；正文要优先引用官方文档、论文、GitHub 仓库和一手发布。

## 页面结构要求

每个专题页建议保持以下结构：

```md
# 专题名称

一句话说明本文解决什么问题。

## 适合读者

## 核心概念

## 工作机制

## 实践步骤

## 常见误区

## 延伸学习

---

## 资料整理状态

<!-- RESOURCES_START -->
自动生成内容
<!-- RESOURCES_END -->
```

脚本只更新 `RESOURCES_START` 到 `RESOURCES_END` 之间的状态内容；如果页面没有该区块，会在末尾追加。该区块不允许输出原始搜索结果链接。

## 更新节奏

| 周期 | 动作 |
|------|------|
| 每日 | 运行 `python scripts/collect.py`，更新资源库和每日报告 |
| 每周 | 人工浏览新增资源，挑选值得吸收进正文的内容 |
| 每月 | 检查导航、学习路径、过时模型/API、失效链接和内容重复 |
| 重大模型发布 | 手动更新模型专区、工具专区和相关进阶专题 |

## 推荐命令

```bash
python scripts/collect.py
npm run build
```

只采集某个专题：

```bash
python scripts/collect.py --topic RAG检索增强 --max-results 10
```

只重建资料整理状态区块：

```bash
python scripts/render_resources.py
```

新增专题后补齐页面：

```bash
python scripts/create_index_pages.py
```

## 判断项目逻辑是否正确

每次改脚本或文档结构后检查：

- `scripts/content_config.py` 中的专题是否都有对应 `docs/<分类>/<专题>/index.md`。
- 首页和侧边栏是否能到达新增专题。
- `data/resources.json` 是否没有重复 URL。
- 新增资源是否有 `provider`、`dedupe_key`、中文标题/摘要和验证信息。
- 每个专题页是否只展示资料整理状态，而不是站外链接列表。
- `npm run build` 是否成功。
- 更新报告是否能说明新增了什么，而不是只生成无上下文链接。
