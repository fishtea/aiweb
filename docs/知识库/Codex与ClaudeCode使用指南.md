# Codex 与 Claude Code 使用指南

这份文档面向日常写代码、改项目、补测试、写文档的人。目标不是把所有参数背下来，而是让你知道什么时候该怎么用、怎么描述任务、怎么少踩坑。

> 适用对象：已经会基本使用命令行、Git、Node/npm，并希望把 AI 编程 Agent 用进真实项目的人。

## 一、如何用好 Codex

Codex 是 OpenAI 的命令行 AI 编程助手。它可以在你的项目目录里读代码、解释代码、修改文件、运行命令、写测试、做代码审查。你可以把它理解成一个能操作本地工作区的编程搭档。

### 1. 安装和配置

常见安装方式是通过 npm 全局安装：

```bash
npm install -g @openai/codex
```

安装后检查版本：

```bash
codex --version
```

登录：

```bash
codex login
```

如果是在无浏览器的服务器上，可以用设备码登录：

```bash
codex login --device-auth
```

如果你想用 API Key 跑本地或自动化任务，可以按 Codex 登录流程选择 API Key 方式。团队或企业环境里，优先确认公司允许哪种登录方式。

常用配置位置：

```text
~/.codex/config.toml       # 个人全局配置
.codex/config.toml         # 项目级配置
AGENTS.md                  # 项目级 Agent 工作说明
```

建议先用默认权限跑一阵，再逐步调整。不要一上来就长期使用完全放开的权限。

### 2. 常用使用方式

#### 直接在终端对话

进入项目目录后启动：

```bash
cd /path/to/project
codex
```

也可以启动时直接给任务：

```bash
codex "阅读这个项目，告诉我启动方式、主要目录结构和测试命令"
```

在一个具体目录启动：

```bash
codex --cd apps/web "帮我梳理这个前端应用的路由结构"
```

#### 让 Codex 读代码、改代码

适合直接交代目标、范围、约束和验收标准：

```text
请阅读 src/auth 目录，修复登录失败时错误提示不准确的问题。

要求：
1. 只改认证相关代码，不做无关重构。
2. 保持现有 UI 文案风格。
3. 如果有测试，补充或更新测试。
4. 完成后运行相关测试，并说明改了哪些文件。
```

让它先理解再动手：

```text
先不要改代码。请先阅读 src/payments 和 tests/payments，说明当前支付回调的处理流程、关键函数、潜在风险点。等我确认后再开始改。
```

#### 让 Codex 运行命令

Codex 可以运行命令，但它通常会受沙箱和审批策略限制。你可以明确告诉它哪些命令是允许的：

```text
请修复这个 TypeScript 报错。可以运行：
- npm run typecheck
- npm test -- auth

不要安装新依赖，不要改 package-lock.json。
```

非交互式运行适合脚本化任务：

```bash
codex exec "检查当前未提交改动，做一次代码审查，只输出高风险问题"
```

诊断本地环境：

```bash
codex doctor
```

### 3. 使用技巧

#### 任务描述要具体

差的描述：

```text
帮我优化一下项目。
```

更好的描述：

```text
请优化 docs 构建速度。先检查 package.json、VitePress 配置和 docs 目录结构。
目标：减少本地 dev 首屏启动时间。
约束：不要替换文档框架，不要改变现有 URL。
完成标准：说明瓶颈、做必要改动，并运行 npm run docs:build 验证。
```

一个好任务通常包含四件事：

- `目标`：最终要解决什么问题。
- `上下文`：相关目录、文件、错误日志、业务背景。
- `约束`：不能改什么、必须遵守什么风格。
- `完成标准`：什么结果才算完成。

#### 复杂任务拆成小步骤

复杂任务不要一次性丢给 Agent。推荐这样拆：

```text
第一步：只做代码阅读，输出现状和风险。
第二步：给出改造方案，不改代码。
第三步：我确认方案后，你再实现。
第四步：补测试并运行验证命令。
第五步：总结 diff 和剩余风险。
```

这样能减少大范围误改，也方便你中途调整方向。

#### 善用文件操作能力

Codex 很适合处理这些文件级工作：

- 批量改 Markdown 文档结构。
- 根据已有模式新增模块。
- 修改配置文件。
- 补测试用例。
- 生成迁移脚本。
- 查找某个函数的调用链。
- 对当前 Git diff 做审查。

示例：

```text
请根据 docs/工具专区 里已有文章的写法，新增一篇 pnpm 使用指南。
要求保留同样的标题层级、代码块风格和结尾检查清单。
```

#### 用 AGENTS.md 固化团队偏好

如果你经常重复告诉 Codex “先读代码”“不要大重构”“改完跑测试”，就应该写进 `AGENTS.md`。

示例：

```md
# AGENTS.md

## 工作约定

- 修改代码前先阅读相关文件，避免凭空改。
- 优先做小而清晰的变更，不做无关重构。
- 修改 TypeScript 后运行 `npm run typecheck`。
- 修改测试相关代码后运行对应测试。
- 最终回复必须说明改动文件、验证命令和未验证风险。
```

Codex 会在启动时读取这些说明。仓库根目录可以放一份，子目录也可以放更具体的规则。

### 4. 典型场景示例

#### 代码审查

```bash
codex "请审查当前未提交改动，重点看 bug、回归风险、测试缺口。不要只总结优点。"
```

#### 修 Bug

```text
用户反馈：上传大文件时进度条会卡在 99%。
请检查 src/upload 相关代码，找到可能原因并修复。
限制：不要改上传协议，不要引入新依赖。
验证：补充或更新测试，并说明如何手动复现。
```

#### 写测试

```text
请为 src/lib/date-range.ts 补单元测试。
重点覆盖：跨月、闰年、空输入、非法日期。
只改测试文件，除非发现实现确实有 bug。
```

#### 文档维护

```text
请阅读 README.md 和 docs/部署指南.md，把部署说明中已经过时的 npm 命令更新为 pnpm。
要求：只改命令和相关说明，不重写整篇文档。
```

#### 重构

```text
请先分析 src/api/client.ts 是否承担了过多职责。
只输出重构计划：目标、拆分方式、风险、测试策略。
不要改代码。
```

## 二、如何用好 Claude Code

Claude Code 是 Anthropic 的编程 Agent，定位也是在终端里理解代码库、编辑文件、运行命令、自动化完成开发任务。它的优势通常体现在长上下文阅读、代码解释、重构方案、测试补充和大型项目梳理上。

### 1. 安装和配置

常见安装方式：

```bash
npm install -g @anthropic-ai/claude-code
```

进入项目并启动：

```bash
cd /path/to/project
claude
```

登录：

```bash
claude login
```

首次进入项目后，可以让 Claude Code 生成项目记忆文件：

```text
/init
```

它通常会生成或更新 `CLAUDE.md`，用于记录项目结构、常用命令、开发约定和注意事项。

### 2. 常用使用方式

#### 交互式对话编程

```bash
claude
```

进入后直接描述任务：

```text
请阅读这个项目，告诉我：
1. 如何本地启动
2. 主要业务模块在哪里
3. 测试命令是什么
4. 新人最应该先读哪些文件
```

也可以直接把任务作为命令参数：

```bash
claude "检查当前 Git diff，帮我找潜在 bug 和测试缺口"
```

#### 让 Claude Code 读项目、理解代码库

适合先让它建立项目地图：

```text
先不要改代码。请阅读 package.json、src、tests 和 docs，输出这个项目的：
- 技术栈
- 启动命令
- 构建命令
- 测试命令
- 核心模块
- 代码风格约定
- 你建议写入 CLAUDE.md 的内容
```

如果项目很大，不要让它一次读全部。给范围：

```text
只看 src/billing、src/db、tests/billing，帮我理解账单生成流程。
重点说明数据从哪里来、在哪里计算、在哪里持久化。
```

#### 自动化重构、写测试

重构前先让它出方案：

```text
请分析 src/components/Table 目录。
目标：把筛选、排序、分页逻辑拆出来，降低组件复杂度。
先输出重构方案和风险，不要改代码。
```

确认后再实现：

```text
按上一步方案实施。
要求：
1. 保持外部 API 不变。
2. 更新或补充测试。
3. 运行相关测试。
4. 最后总结改动和风险。
```

写测试：

```text
请为 src/services/order-service.ts 补测试。
覆盖正常下单、库存不足、重复提交、支付失败四类情况。
尽量沿用 tests 目录里已有测试风格。
```

### 3. 使用技巧

#### 充分利用长上下文，但不要浪费上下文

Claude 擅长读长文件和长链路，但仍然需要你给边界。

推荐：

```text
请重点阅读这些路径：
- src/api
- src/models
- tests/api

不要扫描 node_modules、dist、coverage。
```

不推荐：

```text
把整个项目都看一遍，然后优化一下。
```

#### 给足项目背景

业务背景越清楚，Claude Code 越不容易“技术正确但业务错误”。

示例：

```text
背景：这是一个面向内容团队的 AI 知识库项目。
用户不是工程师，所以错误提示要通俗。
本次目标：优化导入失败时的提示，让用户知道是文件格式、网络问题还是权限问题。
约束：不要改变导入流程，只改错误分类和提示文案。
```

#### 用 CLAUDE.md 定制行为

`CLAUDE.md` 可以记录项目级约定。建议包含：

```md
# CLAUDE.md

## 项目概览

- 这是一个 VitePress 文档站。
- 文档在 `docs/` 目录。
- 中文内容优先使用简洁、实用的表达。

## 常用命令

- `npm run docs:dev`：本地预览
- `npm run docs:build`：构建文档

## 工作规则

- 不要无关重写整篇 Markdown。
- 新增文档要放到对应栏目目录。
- 修改完成后说明新增或修改了哪些文件。
```

如果 Claude Code 总是犯同一种错误，把规则写进去。不要写太虚的规则，比如“写高质量代码”；要写可执行的规则，比如“修改前端组件后运行 `npm run typecheck`”。

### 4. 典型场景示例

#### 快速理解陌生仓库

```text
我是第一次接触这个仓库。请用新人视角解释：
1. 项目解决什么问题
2. 核心目录怎么分
3. 本地如何启动
4. 改一个页面通常要动哪些文件
5. 当前最明显的技术债在哪里
```

#### 生成测试计划

```text
请阅读 src/importer 目录，输出测试计划。
按单元测试、集成测试、手工测试分组。
先不要写代码。
```

#### 批量文档维护

```text
请检查 docs 目录里是否还有 `npm run dev` 的旧命令。
如果当前项目实际使用 `npm run docs:dev`，请只替换相关文档命令，并列出改动文件。
```

#### 大型重构前调研

```text
请分析用户权限相关代码。
目标：后续把 RBAC 和订阅权限拆开。
本轮只做调研，不改代码。
输出：现状、调用链、耦合点、建议拆分步骤、风险。
```

## 三、别人是怎么用的：社区最佳实践

### 1. 常见使用模式

#### 代码审查助手

把 Agent 当成第一轮审查工具，尤其适合检查：

- 空值、边界条件、并发问题。
- 缺少测试。
- 配置改动带来的部署风险。
- 文档和实现不一致。
- 安全问题，比如把 token 写进日志。

示例：

```text
请审查当前分支相对 main 的改动。
重点：bug、回归风险、安全风险、缺少测试。
输出格式：按严重程度排序，每条包含文件位置、问题、建议修复方式。
```

心得：让 Agent “找问题”时，不要让它“评价代码质量”。后者容易空泛，前者更容易产出可行动结果。

#### 自动化测试生成

社区里常见做法是让 Agent 先读已有测试风格，再补测试：

```text
请先阅读 tests 目录中同类模块的测试风格。
然后为 src/cache/session-cache.ts 补测试。
要求：沿用现有测试工具和命名风格，不引入新测试框架。
```

心得：不要只说“补测试”，要列出关键场景。否则 Agent 可能只补快乐路径。

#### 文档生成和维护

适合让 Agent 做这些事：

- 从代码生成 README 的使用示例。
- 把部署步骤整理成文档。
- 更新过时命令。
- 为复杂模块补“开发者说明”。
- 把会议纪要整理成开发任务。

示例：

```text
请根据 scripts/deploy.ts 和 package.json，更新 docs/部署指南.md。
要求：
1. 保留原有文档结构。
2. 命令必须来自 package.json 或脚本实际参数。
3. 不确定的地方标注为“需要人工确认”，不要编。
```

#### Bug 修复流水线

比较稳的流程：

```text
第一步：根据报错和日志定位可能原因，不改代码。
第二步：给出最小修复方案和验证方法。
第三步：实施修复。
第四步：补回归测试。
第五步：运行测试并总结。
```

示例：

```text
报错：TypeError: Cannot read properties of undefined (reading 'id')
发生在用户删除最后一个项目后刷新页面。
请按“定位 -> 方案 -> 修复 -> 测试 -> 总结”的流程处理。
```

#### 重构助手

重构时最怕 Agent 改太多。要明确：

- 外部 API 是否允许变化。
- 是否允许移动文件。
- 是否允许改测试快照。
- 是否允许引入新依赖。
- 哪些目录不能碰。

示例：

```text
请重构 src/report/generator.ts。
目标：降低函数复杂度，拆出纯函数，便于测试。
约束：
- 不改变导出的函数签名。
- 不改变生成报告的字段名。
- 不引入新依赖。
- 每一步保持测试可运行。
```

#### 新功能快速原型

适合低风险、可回滚的功能：

```text
请做一个最小可用版本：
功能：在文档列表页增加按标签筛选。
范围：只改前端状态和列表过滤，不接后端。
完成标准：能在本地预览中选择标签并过滤列表。
```

心得：原型阶段可以让 Agent 快，但合并前必须补测试、收紧边界、做人工审查。

### 2. 与其他工具组合使用

#### Git

常见组合：

```bash
git status
git diff
codex "请审查当前 git diff，找出高风险问题"
claude "请根据当前改动生成 PR 描述，包含测试情况和风险"
```

建议养成习惯：让 Agent 改代码前先确认 `git status`，避免把别人或自己未完成的改动混进去。

#### 测试和类型检查

让 Agent 明确使用项目里的命令：

```text
修改完成后请运行：
- npm run typecheck
- npm test -- --runInBand

如果命令失败，先判断是你的改动导致，还是环境问题。
```

#### Issue / PR 系统

可以把 Issue 内容整理成任务：

```text
请根据下面的 Issue 描述，拆成开发任务：
1. 需要改哪些模块
2. 需要补哪些测试
3. 有哪些风险
4. 推荐实现顺序
```

也可以生成 PR 描述：

```text
请根据当前 git diff 生成 PR 描述：
- 背景
- 改动内容
- 测试情况
- 风险和回滚方式
```

#### 文档站和知识库

适合把 Agent 用作文档维护员：

```text
请检查 docs/知识库 目录中是否有重复主题。
输出建议合并的文章、原因和合并顺序。
不要改文件。
```

### 3. 团队协作中的使用方式

#### 把规则写进项目文件

Codex 用 `AGENTS.md`，Claude Code 用 `CLAUDE.md`。团队可以把这些内容纳入仓库：

```text
项目简介
目录结构
本地启动命令
测试命令
代码风格
文档风格
禁止事项
完成标准
```

不要把密钥、内部账号、生产地址密码写进去。Agent 说明文件应该是开发规则，不是秘密仓库。

#### 统一提示词模板

团队可以沉淀几类模板：

```text
代码审查模板
Bug 修复模板
测试补充模板
文档更新模板
重构计划模板
PR 描述模板
```

示例：Bug 修复模板

```text
请修复以下 Bug：

现象：
复现步骤：
期望行为：
实际行为：
相关文件：
限制：
验证方式：

请先分析原因和方案，等我确认后再改代码。
```

#### 人负责判断，Agent 负责执行

比较稳的分工：

- 人决定目标、范围、优先级。
- Agent 读代码、找调用链、提出方案。
- 人确认方案。
- Agent 实现、补测试、跑命令。
- 人审查 diff、决定是否合并。

不要让 Agent 在没有业务背景的情况下自行决定产品行为，尤其是权限、计费、数据删除、风控、隐私相关功能。

## 四、实战心得和踩坑

### 1. 不要让 Agent 猜

如果你自己都不确定，直接说：

```text
我不确定这个接口是否还在用。请先查调用方和测试，不要直接删除。
```

### 2. 不要一次给太大任务

“重构整个项目”“优化所有文档”“修复所有测试”都太大。先让 Agent 做盘点，再分批执行。

### 3. 明确不能做什么

常见限制要写清楚：

```text
不要引入新依赖。
不要改数据库 schema。
不要改公开 API。
不要重写整篇文档。
不要格式化无关文件。
不要提交代码。
```

### 4. 让 Agent 自己验证

每次改完都要求：

```text
请运行相关测试。如果不能运行，说明原因和你做了哪些静态检查。
```

### 5. 让 Agent 汇报剩余风险

最后回复不要只要“完成了”，要让它说清楚：

```text
请总结：
1. 改了哪些文件
2. 为什么这样改
3. 运行了哪些验证
4. 还有哪些未验证风险
```

### 6. 对生成内容保持审查

Agent 可能会：

- 编造不存在的命令。
- 误解业务规则。
- 改到无关文件。
- 为了测试通过而降低测试价值。
- 过度抽象。
- 忽略性能、安全、权限边界。

所以团队里更推荐“Agent 提效，人做最终判断”的模式。

## 五、推荐日常工作流

### 小任务

```text
目标明确 -> Agent 修改 -> Agent 运行测试 -> 人审查 diff -> 合并
```

### 中等任务

```text
说明背景 -> Agent 调研 -> Agent 出方案 -> 人确认 -> Agent 实现 -> 测试 -> 审查
```

### 大任务

```text
先写计划 -> 拆小任务 -> 每次只改一块 -> 每块都有验证 -> 最后整体回归
```

## 六、快速提示词合集

### 让 Agent 先读项目

```text
请先阅读项目结构，不要改代码。
输出：技术栈、启动命令、测试命令、核心目录、建议优先阅读的文件。
```

### 让 Agent 修 Bug

```text
请修复这个 Bug：
现象：
复现步骤：
相关日志：
约束：
完成标准：

先定位原因并说明方案，确认后再改代码。
```

### 让 Agent 补测试

```text
请为以下模块补测试：
模块：
重点场景：
约束：沿用现有测试框架，不引入新依赖。
完成后运行相关测试并总结。
```

### 让 Agent 做代码审查

```text
请审查当前未提交改动。
只关注：bug、回归风险、安全风险、测试缺口。
按严重程度排序，给出文件位置和修复建议。
```

### 让 Agent 写文档

```text
请根据已有代码和 README，新增一份使用文档。
要求：中文、实用、包含命令示例、不要编造不存在的功能。
不确定的地方标注为需要人工确认。
```
